Git教程

 

前言：本教程并没有赘述Git与svn各项指标性等等的区别，先将svn与git日常版本控制的逻辑流程梳理出来，然后结合教程中不断的深入，相信大家会有自己不同的体会。

 

1. Git与SVN区别

 

SVN版本管理流程

流程概述：

  1. Trunk开发完上线衍生线上版本1.5-tag；
  2. 此时产品在不同时期产生1.6和1.7版本需求，此时需要有两个分支进行不同需求的开发；
  3. Trunk版本此时只作为日常遗留bug修复，如有bug修复需要手动copy同步到1.6和1.7版本；
  4. 在1.6开发的代码同时也需要手动同步到1.7版本中；
  5. 1.5线上版本如有紧急bug修复的，需要手动往Trunk丶1.6丶1.7等所有的版本同步代码；

缺点：

  • 手动同步代码效率极低，容易放错；
  • 代码文件更新同步耗时长；
  • 多个分支并行开发时，管理和开发都存在很大的工作量；

GIT版本管理流程

流程概述：

  1. 当1.6和1.7分支版本并行开发时，此时1.6分支需要发布上线执行：pull push；
  2. 1.6分支发起merge request上线请求，审核通过合并到主干；
  3. 此时1.6分支的代码将进行线上发布，发布完毕1.6分支不会删除并作为test分支修复遗留bug；
  4. 1.6分支开发的内容1.7分支必须要确保都rebase了；
  5. 当1.7提测后1.6分支既可删除不在进行维护；
  6. 如遇紧急BUG则起一个新的bug分支来修复，修复完成后merge到主干并进行发布；
  7. 本地主干pull远程主干，1.7分支rebase本地主干确保1.7分支也包含了最新修复的紧急BUG；

工作区与暂存区

简单说就是你编辑一个文件时是在工作区，执行==add==命令将会放入暂存区，然后执行commit命令将提交到本地分支中。

第一步是用git add把文件添加进去，实际上就是把文件修改添加到暂存区；

第二步是用git commit提交更改，实际上就是把暂存区的所有内容提交到当前分支。

因为我们创建Git版本库时，Git自动为我们创建了唯一一个master分支，所以，现在，git commit就是往master分支上提交更改。

你可以简单理解为，需要提交的文件修改通通放到暂存区，然后，一次性提交暂存区的所有修改。

PS：==commit== 只会提交被==add==的文件到本地分支中去；

6. 分支管理

分支的概念在这里就不赘述，基本和SVN的分支概念是一样的；
其实我们每次提交的commit是连成一条线的，使用HEAD指向当前分支和提交点：

每次提交，master分支都会向前移动一步，这样，随着你不断提交，master分支的线也越来越长；

当我们创建新的分支，例如dev时，Git新建了一个指针叫dev，指向master相同的提交，再把HEAD指向dev，就表示当前分支在dev上：

Git创建一个分支很快，因为除了增加一个dev指针，改改HEAD的指向，工作区的文件都没有任何变化！

不过，从现在开始，对工作区的修改和提交就是针对dev分支了，比如新提交一次后，dev指针往前移动一步，而master指针不变：

多人协助

在实际开发中，我们应该按照几个基本原则进行分支管理：

  1. 首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；
  2. 干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；
  3. 你和你的小伙伴们每个人都在dev分支上干活，甚至每个人都可以在dev基础上有自己的分支，时不时地往dev分支上合并就可以了；
  4. 那如果出现紧急BUG时，通常我们会在master拉取一个BUG分支专门修复上线，完后删除BUG分支，其他分支需要rebase主干；
  5. push远程分支前一定要先pull更新同步一下最新代码。
  6. 如1.6上线后，那么该分支必须删除，而后立马新建一个历史遗留BUG分支来进行修复小BUG，细心的童鞋可能会问为什么不在原始的1.6分支上修复小BUG？其实之前这个问题我也在思考了一下， 这样不是不可以，只是会增加多一步操作，那就是1.6到时候还需要rebase一下master，这样岂不是增加了一些不必要的麻烦嘛。
